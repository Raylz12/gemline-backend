1. Money Loop — Checkout, Escrow & Payouts LIVE

The full buy→ship→deliver→payout money loop is built. Buys create real Stripe manual-capture PaymentIntents — funds are authorized at purchase but only captured when the card is delivered. Sellers can't get paid for cards that never arrive; buyers can't be charged for cancelled orders.

  • Manual-capture checkout — buy creates a pending_payment order + uncaptured PaymentIntent; Payment Element modal collects card details e33de65
  • Escrow states — authorized → captured-on-delivery → payout release; cancel = auth release, no money ever moves
  • Expiry sweep — abandoned pending-payment orders auto-expire and relist (/api/checkout/sweep)
  • Payout-on-delivery — closes the loop; disputes freeze payout (see §2)
  • Address-gated checkout — buyers must have a shipping address; sellers get a SHIP TO panel per sold order 1783c8f

⏳ Final live proof blocked only on the Stripe publishable key + webhook (see §13). All server-side logic verified against live Stripe with cancelled/expired test intents — zero real charges.

2. Orders & Offers Lifecycle LIVE

Offers & Counteroffers 93e1da9

  • Buyers make offers on listings; sellers accept, decline, or counter (counter must beat buyer's number; midpoint pre-filled)
  • Buyer sees a COUNTERED pill with Accept $X / Decline — accept creates a pending-payment order at the counter price through the same escrow checkout

Order Messaging

  • Private buyer↔seller message thread on every order — bubbles, enter-to-send, read receipts, unread badges, deduped notifications

Cancel / Refund / Dispute

  • Buyer requests cancel → seller approves/declines; seller can cancel immediately. Cancel releases the payment auth (no money moved) and auto-relists the card
  • Buyer can dispute post-shipment ("cracked slab" etc.) → payout frozen, both parties notified, admin resolves

Status Timeline

  • Visual Placed → Paid → Shipped → Delivered → Complete strip with real dates on every order; cancelled/refunded/disputed states styled; tracking-number entry + tracking link on shipment

  • Sell page stat cards: For Sale (count + value), Open Offers, Sold (count + gross)

12. How Everything Was Verified

Every claimed feature passed three mandatory layers:

  • 1. Source — code grep confirming the implementation is committed and built
  • 2. Prod probe — live API/DB calls exercising the real flow on gemlinecards.com (e.g., offer $30 → counter $40 → accept → order → message → cancel → relist; report → dedupe → 400 on bad reason; suspend → login 403 → reinstate; flag off → 503 → flag on)

All test data scrubbed after each batch: orders, escrow rows, messages, disputes, reports, blocks, watches, notifications, throwaway users. Stripe test intents cancelled or left unconfirmed (auto-expire) — zero real charges. The 287K-card catalog, real users, and portfolios were never modified — all migrations additive.
